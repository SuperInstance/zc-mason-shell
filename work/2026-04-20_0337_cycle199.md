## Analysis of Existing Tests
From `test_deadband.py`, I observed the following test patterns:
1. **Basic Protocol Flow**: Tests P0→P1→P2 progression with valid inputs.
2. **Negative Space Detection**: Tests that invalid actions (P0 violations) are rejected.
3. **Channel Persistence**: Tests that once a safe channel is found (P1), it can be optimized (P2).
4. **State Reset**: Tests that resetting the room clears the protocol state.
5. **Tile Submission**: Tests that valid interactions produce training tiles.

The existing tests cover the primary happy paths and some obvious error cases (e.g., skipping P1). However, they lack edge cases around:
- Boundary conditions for channel discovery
- Concurrency or rapid state transitions
- Malformed or extreme input values
- Resource exhaustion (e.g., memory, tile limits)
- Protocol state corruption and recovery

## 5 Additional Edge-Case Tests for DeadbandRoom

Based on the Deadband Protocol's strict priority (P0→P1→P2) and the room's role in generating training tiles, I propose the following edge-case tests:

### Test 1: **Empty Channel Set at P1**
**Scenario**: The agent successfully maps negative space (P0) but the environment returns zero safe channels at P1.  
**Expected**: Room should transition to a "deadlock" state, emit a specific error tile (P1_FAILURE), and not allow progression to P2. The agent must be able to reset or receive new environment data.  
**Rationale**: Tests robustness when no viable path exists after avoiding obstacles.

### Test 2: **Maximum Tile Flood**
**Scenario**: Agent rapidly submits 1000 valid interactions (P0→P1→P2 cycles) within a short timeframe.  
**Expected**: Room should throttle or queue tile generation, not crash. Tile storage/memory should remain stable. Later tiles may be delayed or batched, but protocol state must remain correct for each interaction.  
**Rationale**: Tests system resilience under high load and prevents resource exhaustion.

### Test 3: **Malformed Action Payload**
**Scenario**: Agent sends a P1 action with a valid channel ID but an extrinsically malformed payload (e.g., channel ID is a negative integer, a string exceeding 1024 characters, or a nested JSON object beyond spec).  
**Expected**: Room should reject the action, remain in P1 state, and emit an error tile indicating invalid input. It should not crash or corrupt internal state.  
**Rationale**: Ensures input validation and graceful failure handling.

### Test 4: **State Rollback After P2 Failure**
**Scenario**: Agent completes P0 and P1, begins P2 optimization, then submits a P2 action that fails (e.g., optimization exceeds limits).  
**Expected**: Room should revert to P1 state (last known safe channel), not to P0 or a corrupted state. A failure tile should be generated. The agent can then attempt a different P2 action or reset.  
**Rationale**: Tests that the protocol maintains fallback integrity and does not leave the agent stranded.

### Test 5: **Concurrent Protocol Sessions**
**Scenario**: Two independent agents (simulated) interact with the same DeadbandRoom instance simultaneously, each at different protocol phases (e.g., Agent A at P0, Agent B at P2).  
**Expected**: Room should handle concurrent sessions without cross-talk. Each agent's state should be isolated. Tiles should be correctly attributed to each session. No deadlocks or race conditions.  
**Rationale**: Tests thread-safety and session isolation, which is critical for fleet-scale training.
