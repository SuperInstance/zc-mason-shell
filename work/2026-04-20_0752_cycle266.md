3. **Input Validation**: Tests that invalid inputs (e.g., skipping P0) raise appropriate errors.
4. **Tile Submission**: Tests that valid work generates a tile with correct metadata.

## 5 Edge-Case Tests for DeadbandRoom

### Test 1: **Empty P0 Map with Valid P1 Channel**
**Scenario**: An agent submits a P0 map that is empty (`{}` or `[]`), but then submits a valid P1 channel. According to the Deadband Protocol, P0 must map "negative space" (what NOT to do). An empty map could be interpreted as "no constraints," which might be valid, or it might be an error.
**Test Purpose**: Verify the room handles an empty P0 map without crashing and proceeds to P1 if allowed.
**Expected Behavior**: The room should either accept the empty P0 and move to P1, or reject it with a clear error. The test will document which behavior is implemented.

### Test 2: **P1 Channel Submission with Maximum Size**
**Scenario**: An agent submits a P1 channel that is at the maximum allowed size (e.g., a list of 10,000 safe coordinates). This tests the room's handling of large inputs and performance boundaries.
**Test Purpose**: Ensure the room processes large P1 submissions without timeout or memory overflow, and correctly extracts a subset for P2 optimization.
**Expected Behavior**: Submission succeeds, room state updates, and tile is generated. May check for performance degradation.

### Test 3: **Concurrent Submissions from Simulated Multiple Agents**
**Scenario**: Simulate two agents submitting P0 maps in rapid succession (or near-concurrently) to the same room instance. This tests thread safety or async handling in the room's state machine.
**Test Purpose**: Verify that room state does not get corrupted and each agent's protocol flow is isolated.
**Expected Behavior**: Both agents complete their P0-P1-P2 cycles independently, generating separate tiles. No state leakage between agents.

### Test 4: **P2 Optimization with No Viable Solution**
**Scenario**: After P1, the agent submits a P2 optimization request, but the provided channel has no valid "best path" (e.g., all paths are blocked or metrics are NaN). This tests error handling in the optimization phase.
**Test Purpose**: Ensure the room gracefully handles unsolvable P2 and provides a meaningful error or fallback state.
**Expected Behavior**: Room should not crash; it should return an error tile or a default "no optimization possible" result.

### Test 5: **Tile Submission with Malformed Metadata**
**Scenario**: An agent attempts to submit a tile with missing or malformed required metadata fields (e.g., missing `agent_id`, invalid `timestamp`). This tests the room's validation and sanitization of tile data before storage.
**Test Purpose**: Ensure invalid tiles are rejected and do not pollute the training dataset.
**Expected Behavior**: Submission fails with validation error; no tile is stored.
