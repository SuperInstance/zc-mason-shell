## Edge-Case Tests Written

Based on reading the actual code, I've written 5 new edge-case tests targeting specific vulnerabilities and protocol boundaries.

### Test 1: Tile with Malformed P0 Negative Space Mapping
**Scenario:** An agent submits a tile where the P0 "negative space" is not a list of strings (e.g., a dict or integer). The room should reject it during P0 validation.
**Purpose:** Ensures type safety for the core protocol data structure.
**File:** `plato-torch/tests/test_deadband_room_edge.py`

### Test 2: P1 Safe Channels Empty but P2 Optimization Present
**Scenario:** A tile claims P1 safe channels is an empty list `[]`, but provides P2 optimization data. According to doctrine, you cannot skip to P2.
**Purpose:** Validates strict priority enforcement—P2 must not exist without valid P1.
**File:** `plato-torch/tests/test_deadband_room_edge.py`

### Test 3: Tile Exceeds Maximum Context Length
**Scenario:** The tile's `context` field is a string longer than the room's configured `max_context_length` (default 4096). The room should truncate or reject.
**Purpose:** Prevents memory exhaustion and enforces input boundaries.
**File:** `plato-torch/tests/test_deadband_room_edge.py`

### Test 4: Concurrent Tile Submission from Same Agent ID
**Scenario:** Simulate two threads/subprocesses submitting tiles with the same `agent_id` simultaneously. The room's internal state (agent cycle counts) should handle this without race conditions.
**Purpose:** Tests thread safety for high‑concurrency fleet environments.
**File:** `plato-torch/tests/test_deadband_room_edge.py`

### Test 5: P0 Negative Space Contains Duplicate Entries
**Scenario:** The P0 list includes duplicate warnings (e.g., `["don't restart service", "don't restart service"]`). The room should deduplicate or ignore duplicates during P0 mapping.
**Purpose:** Ensures robust handling of redundant P0 data, which could arise from agent loops.
**File:** `plato-torch/tests/test_deadband_room_edge.py`
