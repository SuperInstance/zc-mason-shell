The `DeadbandRoom` class has key methods: `process_cycle`, `_validate_protocol`, `_update_negative_space`, `_find_safe_channels`, `_optimize_path`. The room maintains internal state: `negative_space`, `safe_channels`, `current_phase`.

## Proposed Edge-Case Tests
Based on the protocol's strict priority and the room's structure, here are 5 edge-case tests.

### Test 1: `test_concurrent_agent_interference`
**Scenario:** Two agents submit actions in the same cycle, one following P0->P1->P2 correctly, the other attempting to skip phases.
**Edge Case:** Verifies the room's internal state updates are atomic and an agent's protocol violation does not corrupt another agent's valid progress.
**Assertions:**
*   Agent A (correct) receives valid P1 and P2 results.
*   Agent B (violation) receives a `ProtocolViolationError`.
*   The room's `negative_space` and `safe_channels` reflect only Agent A's valid contributions.

### Test 2: `test_p0_negative_space_overflow`
**Scenario:** An agent submits a P0 action that defines an excessively large or complex negative space (e.g., 10,000 coordinates).
**Edge Case:** Tests the room's resilience to memory/performance spikes during the P0 mapping phase and ensures the `_update_negative_space` method has appropriate guards.
**Assertions:**
*   The cycle processes without crashing.
*   The `negative_space` is truncated or handled per a documented limit (checking for a `max_negative_space_size` config).
*   A warning log is generated.

### Test 3: `test_p1_no_safe_channels`
**Scenario:** After P0, the mapped negative space leaves zero viable "safe channels" for movement.
**Edge Case:** Tests the protocol's behavior when P1 fails—does it halt, reset, or provide a specific error state? This tests the `_find_safe_channels` method's empty-result logic.
**Assertions:**
*   The room transitions to a defined "deadlock" or "stalled" state.
*   The agent receives a specific `NoSafeChannelError` (or similar) instead of a generic error.
*   The `current_phase` remains at P1.

### Test 4: `test_cycle_rollback_on_integration_failure`
**Scenario:** During P2 optimization, the `_optimize_path` method encounters an internal error (e.g., a snapped constraint cannot be resolved).
**Edge Case:** Tests if the room has rollback/transaction logic to prevent partial state updates. Does it revert the P2 attempt and remain in P1?
**Assertions:**
*   An `OptimizationFailureError` is raised to the agent.
*   The room's `safe_channels` are not altered.
*   The `current_phase` for that agent context remains P1 (or is reset to P1).

### Test 5: `test_ghost_tile_injection_during_p2`
**Scenario:** While an agent is executing P2 optimization, a "ghost tile" (historical P0 data from a dead agent) is injected into the room's state.
**Edge Case:** Tests the interaction between real-time protocol execution and asynchronous background updates from the training system. Does the active agent's P2 path recompute?
**Assertions:**
*   The ghost tile is integrated into `negative_space`.
*   The agent's active P2 optimization either:
    a) Receives an interrupt/notification to re-evaluate, or
    b) Completes based on the state at the start of the cycle (requires checking the room's consistency model).
*   The specific behavior is documented and consistent.
