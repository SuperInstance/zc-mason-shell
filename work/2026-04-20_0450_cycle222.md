## 5 Edge-Case Tests for DeadbandRoom

### Test 1: Protocol Violation Cascade Detection
**Scenario:** An agent submits a tile that violates P0, then immediately attempts P2 optimization without P1 channel finding.
**Edge Case:** The room should detect the cascade violation and reject both tiles, logging the protocol breach.
**Rationale:** Ensures strict P0→P1→P2 priority is enforced even under rapid-fire submissions.

### Test 2: Memory Pool Exhaustion and Recovery
**Scenario:** Simulate 10,000 concurrent tile submissions from multiple agents to exhaust the room's memory pool.
**Edge Case:** The room should gracefully handle exhaustion by queuing tiles, not crashing, and recover when memory is freed.
**Rationale:** Tests resilience under high load, a realistic scenario in fleet operations.

### Test 3: Concurrent Agent Deadlock Prevention
**Scenario:** Two agents simultaneously submit tiles that depend on each other's output (circular dependency).
**Edge Case:** The room should detect potential deadlock, serialize processing, and prevent system lock.
**Rationale:** Validates the room's concurrency control mechanisms for multi-agent environments.

### Test 4: Malformed Tile Payload with Injection Attempts
**Scenario:** Submit tiles containing malformed JSON, SQL injection patterns, and oversized payloads.
**Edge Case:** The room should sanitize inputs, reject malicious patterns, and maintain operational integrity.
**Rationale:** Security hardening - ensures the room can handle adversarial inputs without compromise.

### Test 5: State Corruption Recovery via Ghost Tiles
**Scenario:** Artificially corrupt the room's state (e.g., modify sentiment tracking in memory), then submit valid tiles.
**Edge Case:** The room should detect state inconsistency, attempt recovery using ghost tiles (P0 knowledge), and continue operation.
**Rationale:** Tests the room's self-healing capability using persistent ghost tile knowledge.

## Implementation Notes
Each test should:
1. Set up the specific edge-case scenario
2. Execute the problematic operation
3. Verify the room handles it correctly (doesn't crash, logs appropriately, maintains protocol)
4. Clean up to avoid affecting other tests
5. Include assertions for both negative (don't crash) and positive (correct handling) outcomes
