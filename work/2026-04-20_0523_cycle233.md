## 5 Additional Edge-Case Tests

### Test 1: **Malformed Priority Chains**
**Scenario:** A tile claims to follow P0→P1→P2 chain but contains contradictory constraints (e.g., P0 says "avoid X" but P2 optimizes using X).
**Expected:** Room should reject the tile or flag it for manual review.
**Rationale:** Ensures protocol integrity across priority levels.

### Test 2: **Ghost Tile Injection**
**Scenario:** Simulate a "ghost tile" from a terminated agent with only P0 (negative space) knowledge.
**Expected:** Room should accept the P0-only tile and incorporate it as foundational negative space, but not allow it to influence P1/P2 optimization.
**Rationale:** Validates the "ghost tiles" architecture feature mentioned in FLEET-CONTEXT.md.

### Test 3: **Concurrent Tile Submission Race**
**Scenario:** Two simulated agents submit tiles simultaneously with conflicting P1 safe channels.
**Expected:** Room should handle race conditions (via locking or conflict resolution) and maintain consistent state without crashing.
**Rationale:** Tests real-world multi-agent interaction scenarios.

### Test 4: **Priority Boundary Violation**
**Scenario:** A tile attempts to skip from P0 directly to P2 optimization, violating "Never skip to P2" doctrine.
**Expected:** Room should reject the tile or automatically demote P2 content to P1 for reevaluation.
**Rationale:** Enforces strict priority sequencing from fleet doctrine.

### Test 5: **Room Capacity and Memory Limits**
**Scenario:** Submit an excessive number of tiles (e.g., 10,000) or an extremely large tile (>10MB).
**Expected:** Room should implement graceful degradation—queue management for volume, size limits for individual tiles—without losing previously accepted tiles.
**Rationale:** Ensures robustness under load, important for fleet-scale operation.
