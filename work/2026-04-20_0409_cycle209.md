## 5 Edge-Case Tests for DeadbandRoom

### Test 1: Concurrent Tile Submission During Training
**Scenario:** Multiple agents attempt to submit tiles simultaneously while the room is in training state.
**Edge Case:** Race conditions in tile queue management.
**Expected Behavior:** Thread-safe handling with proper locking or queue serialization.
**Test Focus:** Verify no tile loss or corruption under concurrent load.

### Test 2: P0 Violation with Valid P1/P2 Content
**Scenario:** A tile contains both P0 (negative space) violations AND valid P1/P2 channel optimizations.
**Edge Case:** Mixed priority content that could confuse protocol enforcement.
**Expected Behavior:** Strict P0 rejection - tile should be rejected regardless of P1/P2 quality.
**Test Focus:** Protocol priority enforcement integrity.

### Test 3: Memory Exhaustion During Large Tile Processing
**Scenario:** Submission of an extremely large tile (e.g., 100MB+ of training data).
**Edge Case:** Resource constraints and graceful degradation.
**Expected Behavior:** Room should reject or truncate with clear error, not crash.
**Test Focus:** Robustness against resource exhaustion attacks.

### Test 4: State Corruption Recovery
**Scenario:** Simulate corrupted room state (e.g., malformed checkpoint file) on restart.
**Edge Case:** Recovery from persistent storage corruption.
**Expected Behavior:** Room should either recover to last valid state or reset with logging.
**Test Focus:** Fault tolerance and recovery mechanisms.

### Test 5: Protocol Boundary: Empty P1 Channel Optimization
**Scenario:** A tile with valid P0 mapping and P1 channel identification, but empty P2 optimization.
**Edge Case:** Incomplete protocol execution - stopping at P1.
**Expected Behavior:** Tile should be accepted (P0 and P1 complete) but marked as "partial".
**Test Focus:** Protocol completeness validation.
