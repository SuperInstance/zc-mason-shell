## Proposed Edge-Case Tests
Based on the Deadband Protocol doctrine (P0→P1→P2), I will write tests that probe boundaries and failure modes within the established safe channels.

### Test 1: Tile Priority Collision (P0 → P1 Boundary)
**Scenario:** Two tiles with identical `tile_id` but different content are added. The spec suggests tiles are unique by ID. Test the system's handling of this collision.
**Purpose:** Map negative space (P0) — what happens when uniqueness constraint is violated? Does it reject, overwrite, or error?
**File:** `test_deadband_room.py`

### Test 2: Massive Tile Batch with Memory Limits (P1 Channel Stress)
**Scenario:** Add 10,000 tiles in a loop, simulating a high-volume training session. Monitor memory/performance.
**Purpose:** Find safe channels (P1) — does the room handle scale within expected limits? Does it throttle, crash, or slow gracefully?
**File:** `test_deadband_room.py`

### Test 3: Corrupted Tile JSON Structure (P0 Negative Space)
**Scenario:** Provide a tile with valid JSON but deeply nested/invalid fields (e.g., cyclical references, extreme depth).
**Purpose:** Map negative space (P0) — how does the room's validation handle structurally valid but semantically corrupt data?
**File:** `test_deadband_room.py`

### Test 4: Ensemble Export with Partial Training (P1 → P2 Transition)
**Scenario:** Call `export_ensign` after only 1 training step (minimal data). Then call it after 100 steps.
**Purpose:** Optimize within channels (P2) — does the export quality or behavior change with training depth? Is there a minimum viable step count?
**File:** `test_deadband_room.py`

### Test 5: Concurrent Tile Addition (Race Condition)
**Scenario:** Simulate two threads/processes adding tiles simultaneously to the same room instance.
**Purpose:** Map negative space (P0) — is the room thread-safe? Does it have locking mechanisms or does it break?
**File:** `test_deadband_room.py`
