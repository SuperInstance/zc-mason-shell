2. Tile submission with valid priority flow (P0→P1→P2)
3. Rejection of tiles that skip priority levels

5. Room state persistence across sessions
6. Concurrent tile submissions from multiple agents
7. Room capacity limits

The tests use pytest fixtures and mock the underlying tile storage. Coverage appears strong for happy paths and common failure modes.

## Identified Edge Cases
Based on reading the DeadbandRoom implementation, I identified these untested scenarios:

1. **Priority inversion within same cycle** - What if P2 tile arrives before P0 in same batch?
2. **Empty constraint sets** - P0, P1, or P2 constraints being empty lists
3. **Duplicate constraint definitions** - Same constraint appearing in multiple priority levels
4. **Extremely large tile payloads** - Testing memory/performance boundaries
5. **Mixed priority tiles from single agent** - Agent submitting P0 and P2 tiles simultaneously
