## Analysis of DeadbandRoom Implementation
Based on reading the Rust source, the `DeadbandRoom` preset implements the fleet's Deadband Protocol (P0, P1, P2) as a training environment. Key components:
- `DeadbandRoom::new()` initializes with a `deadband_threshold` (f64) and `channels` (Vec<Channel>)
- `Channel` struct has `id`, `safe_min`, `safe_max`, `optimal_point`
- Core method `apply_protocol(&self, input: f64) -> Action` returns an `Action` enum (`Avoid`, `Navigate`, `Optimize`)
- Logic: if input.abs() < threshold → P0 `Avoid`, else find first channel where input in range → P1 `Navigate`, else → P2 `Optimize` to nearest channel optimal point
- Error handling: returns `Action::Avoid` on NaN input

### Test 2: Overlapping Safe Channels
**Scenario:** Two channels with overlapping safe ranges (e.g., Channel A: 1.0-4.0, Channel B: 3.0-6.0). Input = 3.5 lies in both.
**Edge Case:** Which channel does P1 `Navigate` choose? First match in iteration order.
**Expected Behavior:** Protocol should be deterministic—pick first matching channel. This may affect training stability if order isn't guaranteed.
```rust
#[test]

    let channels = vec![
        Channel { id: 1, safe_min: 1.0, safe_max: 4.0, optimal_point: 2.5 },
        Channel { id: 2, safe_min: 3.0, safe_max: 6.0, optimal_point: 4.5 },

    let room = DeadbandRoom::new(0.5, channels);

        Action::Navigate(channel_id, _) => assert_eq!(channel_id, 1), // first channel
```

### Test 3: Deadband Threshold Exactly at Channel Boundary
**Scenario:** Deadband threshold = 2.0. Channel safe range starts at 2.0. Input = 2.0.
**Edge Case:** Is input considered inside deadband (P0 `Avoid`) or inside channel (P1 `Navigate`)? Depends on inequality strictness.
**Expected Behavior:** Code uses `<` for deadband check (`input.abs() < threshold`), so input.abs() == threshold is NOT in deadband. Then channel uses `>= safe_min && <= safe_max` (likely inclusive). So input 2.0 with threshold 2.0 should go P1.
```rust
#[test]
fn test_threshold_at_channel_boundary() {
    let channels = vec![Channel { id: 1, safe_min: 2.0, safe_max: 5.0, optimal_point: 3.5 }];
    let room = DeadbandRoom::new(2.0, channels); // threshold equals channel min
    let action = room.apply_protocol(2.0);
    // input.abs() == threshold, not <, so not P0.
    // input >= safe_min (2.0 >= 2.0) true, so P1 Navigate.
    match action {
        Action::Navigate(channel_id, _) => assert_eq!(channel_id, 1),
        _ => panic!("Should be Navigate, got {:?}", action),
    }

```

### Test 4: Negative Input with Asymmetric Channels
**Scenario:** Channels defined only for positive ranges (e.g., 1.0-3.0). Input = -2.0 (outside deadband, negative).
**Edge Case:** No channel matches negative input. P2 `Optimize` must find nearest channel based on absolute distance? Likely uses `optimal_point` difference.
**Expected Behavior:** Should compute distance to each channel's optimal_point, pick minimal absolute difference. For -2.0, nearest optimal_point might be 2.5 (id 1). Action should be `Optimize` toward that channel's optimal point.
```rust
#[test]
fn test_negative_input_no_negative_channels() {
    let channels = vec![
        Channel { id: 1, safe_min: 1.0, safe_max: 3.0, optimal_point: 2.0 },
        Channel { id: 2, safe_min: 4.0, safe_max: 6.0, optimal_point: 5.0 },
    ];
    let room = DeadbandRoom::new(0.5, channels);
    let action = room.apply_protocol(-2.0);
    // Not in deadband (abs 2.0 > 0.5). No channel safe range includes -2.0.
    // P2: find channel with optimal_point closest to -2.0 (abs diff).
    // |2.0 - (-2.0)| = 4.0, |5.0 - (-2.0)| = 7.0, so channel 1 is nearest.
    match action {
        Action::Optimize(channel_id, target) => {
            assert_eq!(channel_id, 1);
            assert_eq!(target, 2.0); // optimal_point of channel 1
        },
        _ => panic!("Should be Optimize"),
    }
}
```
