## Analysis of Existing Tests
From `test_deadband_room.py`, I observe the following test patterns:
1. **Protocol Phase Validation**: Tests that the room correctly identifies P0, P1, P2 states.
2. **Tile Acceptance Logic**: Tests that valid tiles are accepted and invalid ones rejected.
3. **State Transitions**: Tests that the room moves between phases based on tile content.
4. **Error Handling**: Tests for malformed tiles, missing fields, and invalid JSON.
5. **Boundary Conditions**: Tests for empty tile lists, max tile limits, and duplicate submissions.

The existing tests cover the primary happy paths and some basic error cases. Edge cases appear to be areas where behavior under stress, race conditions, or ambiguous inputs is not yet validated.

## 5 Additional Edge-Case Tests

Based on the Deadband Protocol's strict priority (P0 → P1 → P2, never skip) and the room's role in training ensigns, I propose the following edge-case tests:

### 1. **Test: Concurrent Tile Submission Race Condition**
**Scenario:** Simulate two agents submitting tiles for the same cycle simultaneously.
**Edge:** The room's tile acceptance logic must handle concurrent writes without corruption or double-counting.
**Validation:** Ensure total tile count matches submissions, no tiles are lost, and phase calculation remains consistent.

### 2. **Test: P0 Tile with Ambiguous Negative Space**
**Scenario:** Submit a P0 tile where the "negative_space" field contains contradictory or self-referential constraints (e.g., "avoid avoiding").
**Edge:** The room must parse and store the tile without crashing, even if the content is logically ambiguous. Ensign training later may handle interpretation.
**Validation:** Tile is accepted, stored, and does not break phase detection.

### 3. **Test: Maximum Tile Burst Followed by Immediate Phase Shift**
**Scenario:** Submit the maximum allowed tiles per cycle (if a limit exists), then immediately submit a tile that should trigger a phase transition (e.g., P0 complete → P1).
**Edge:** The room must process the burst, update state, and correctly evaluate the phase-shift tile under the new state.
**Validation:** Phase transitions correctly after burst, and the new tile is evaluated in the proper context.

### 4. **Test: Malformed Priority Field (Non-Integer)**
**Scenario:** Submit a tile with `priority: "P0"` (string) instead of `priority: 0` (integer).
**Edge:** The room's validation should catch type mismatches and reject the tile with a clear error, not crash.
**Validation:** Tile is rejected, error message indicates invalid priority type, room state remains unchanged.

### 5. **Test: Historical Tile Injection (Out-of-Sequence Cycle)**
**Scenario:** Attempt to submit a tile with a `cycle` field far in the past or future relative to the room's current cycle.
**Edge:** The room must decide whether to accept, reject, or adjust the tile's cycle. This tests temporal consistency in training data.
**Validation:** Based on the room's design, it should likely reject or normalize the cycle to maintain a contiguous timeline.
