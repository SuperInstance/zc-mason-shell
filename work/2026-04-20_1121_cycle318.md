These are functional but lack edge cases around state persistence, malformed input, concurrency, and protocol boundary conditions.

## Proposed Edge-Case Tests
Based on the DeadbandRoom implementation, here are 5 additional edge-case tests:

### 1. Test: `test_validate_tile_empty_content`
**Scenario:** A tile is submitted with `content` field as empty string or whitespace-only.
**Expected:** Rejected. The room should require substantive content for training value.
**Rationale:** Prevents noise tiles from polluting training data.

### 2. Test: `test_validate_tile_missing_priority_field`
**Scenario:** A tile JSON is missing the `priority` field entirely (not just malformed).
**Expected:** Rejected. The protocol is priority-driven; a missing priority is invalid.
**Rationale:** Ensures robust parsing and protocol adherence.

### 3. Test: `test_p1_channel_conflict_detection`
**Scenario:** Two agents submit P1 tiles that define contradictory "safe channels" for the same context.
**Expected:** The room should accept both (P1 is about exploration) but flag a conflict in its internal state for review.
**Rationale:** Safe channel mapping isn't necessarily unique; conflict detection is a room intelligence feature.

### 4. Test: `test_p2_optimization_without_p1_context`
**Scenario:** A P2 tile references a "channel_id" from a P1 tile that does not exist in the room's state.
**Expected:** Rejected. P2 must build upon an established P1 channel.
**Rationale:** Enforces strict protocol sequencing; prevents orphaned optimizations.

### 5. Test: `test_tile_idempotency_rejection`
**Scenario:** The same tile (identical hash) is submitted twice by the same agent.
**Expected:** Rejected on the second submission. Prevents duplicate training data.
**Rationale:** Idempotency handling conserves storage and processing.
