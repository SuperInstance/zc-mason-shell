## 5 Edge-Case Tests for DeadbandRoom

### Test 1: Concurrent Tile Submission from Multiple Agents
**Scenario:** Simulate two agents submitting tiles simultaneously to the same room.
**Edge:** Race conditions in tile queue, state corruption.
**Assertions:** All tiles are processed, no data loss, room state remains consistent.
**P0 Check:** Ensure no deadlock or dropped tiles.

### Test 2: Malformed Tile Payload (Invalid JSON Structure)
**Scenario:** Submit a tile with a payload that is not valid JSON or missing required fields.
**Edge:** Parser errors, room crash, silent failure.
**Assertions:** Room rejects tile with clear error, does not crash, logs the incident.
**P0 Check:** Room remains operational.

### Test 3: Protocol Violation During P0 Phase (Skip to P2)
**Scenario:** Agent attempts to submit a P2 optimization tile while room is in P0 mapping phase.
**Edge:** Protocol enforcement, state validation.
**Assertions:** Room rejects tile, increments violation counter, may apply penalty (e.g., cooldown).
**P0 Check:** Strict priority is enforced.

### Test 4: Room State Persistence Across Restarts
**Scenario:** Simulate server restart, verify room reloads previous phase, tile queue, and agent states.
**Edge:** Data serialization/deserialization errors, state loss.
**Assertions:** Phase, queue, and violation counts are restored exactly.
**P0 Check:** No data corruption.

### Test 5: Maximum Tile Queue Capacity and Overflow
**Scenario:** Flood room with tiles exceeding configured queue limit (e.g., 1000 tiles).
**Edge:** Memory exhaustion, performance degradation, tile rejection logic.
**Assertions:** Room enforces limit, rejects excess tiles gracefully, maintains performance.
**P0 Check:** System does not crash or become unresponsive.
