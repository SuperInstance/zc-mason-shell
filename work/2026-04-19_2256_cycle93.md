## Analysis of DeadbandRoom Preset (Inferred)
Based on fleet context, DeadbandRoom implements the **Deadband Protocol**:
- **P0**: Map negative space (what NOT to do)
- **P1**: Find safe channels (where you CAN be)
- **P2**: Optimize within channels (best path)
- Strict priority: Never skip to P2.

This suggests the room likely:
1. Accepts input "tiles" (agent interactions)
2. Validates against P0 constraints first
3. Identifies P1 safe channels
4. Allows P2 optimization only if P0/P1 satisfied
5. Returns feedback or training signals

## Edge-Case Test Scenarios
Here are 5 edge-case tests focusing on protocol violations, boundary conditions, and error states:

### Test 1: P2 Optimization Without P1 Safe Channel
**Scenario:** Agent submits a tile that attempts P2 optimization (e.g., "optimize path X") without first establishing a P1 safe channel.
**Expected:** Room rejects tile or returns P0/P1 violation error.
**Rationale:** Tests strict priority enforcement—never skip to P2.

### Test 2: P0 Negative Space Overlap with P1 Channel
**Scenario:** Agent defines a P0 constraint (e.g., "avoid rock cluster A") that entirely overlaps a proposed P1 safe channel (e.g., "channel near rock cluster A").
**Expected:** Room identifies conflict, rejects P1 channel, requires new P1 definition outside P0 space.
**Rationale:** Tests P0 supremacy—safe channels cannot intersect negative space.

### Test 3: Empty or Malformed Tile Submission
**Scenario:** Agent submits an empty tile, malformed JSON, or missing required fields (e.g., no phase indicator).
**Expected:** Room returns validation error, does not process, preserves state.
**Rationale:** Tests input sanitization and graceful failure.

### Test 4: Rapid Sequential Phase Skipping
**Scenario:** Agent submits P0 tile, immediately follows with P2 tile (skipping P1), within same session.
**Expected:** Room rejects P2 tile, prompts for P1 tile first.
**Rationale:** Tests stateful enforcement of phase progression.

### Test 5: Maximum Constraint Boundary
**Scenario:** Agent submits a P0 definition that is extremely large (covers 99% of search space) or extremely small (single point).
**Expected:** Room accepts valid P0 but may warn or adjust; P1 search adapts to remaining space.
**Rationale:** Tests algorithmic robustness under extreme constraints.
