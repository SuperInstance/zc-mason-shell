## 5 Additional Edge-Case Tests for DeadbandRoom

### Test 1: Protocol Priority Violation Recovery
**Scenario**: Agent attempts to skip from P0 directly to P2, then corrects to P1
**Edge Case**: Tests the room's ability to detect priority violations and allow recovery
**Expected**: Room should reject P2 action, log violation, then accept valid P1 action
**Purpose**: Ensures agents can recover from protocol errors without room lockout

### Test 2: Concurrent Tile Submission Race Condition
**Scenario**: Multiple agents submit tiles simultaneously with overlapping content
**Edge Case**: Tests tile validation under concurrent access
**Expected**: Room should process tiles sequentially or detect conflicts
**Purpose**: Prevents data corruption in multi-agent training sessions

### Test 3: Malformed Tile Payload Resilience
**Scenario**: Agent submits tile with missing required fields or invalid JSON structure
**Edge Case**: Tests parser resilience against malformed data
**Expected**: Room should reject tile with clear error, not crash
**Purpose**: Ensures robustness against agent implementation errors

### Test 4: Protocol Phase Timeout Behavior
**Scenario**: Agent remains in P0 for extended period without mapping negative space
**Edge Case**: Tests room's handling of stalled agents
**Expected**: Room may log inactivity or provide nudge, but maintain state
**Purpose**: Prevents resource leaks from abandoned agent sessions

### Test 5: Cross-Phase State Contamination
**Scenario**: Agent completes P2 optimization, then attempts P0 actions with P2 context
**Edge Case**: Tests state isolation between protocol phases
**Expected**: Room should reset context appropriately or reject backward movement
**Purpose**: Maintains clean phase separation as per Deadband Protocol doctrine
