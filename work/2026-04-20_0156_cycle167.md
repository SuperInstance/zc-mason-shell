## 5 Additional Edge-Case Tests

Based on the Deadband Protocol implementation and fleet context, here are 5 edge cases not covered by existing tests:

### 1. **Test: P0 Mapping with Conflicting Constraints**
**Scenario:** Agent receives constraints that directly contradict each other (e.g., "avoid A" and "must include A").
**Expected:** Room should detect contradiction, log P0 violation, and return safe empty state rather than crashing.
**Rationale:** Real-world constraints from multiple sources may conflict; system should handle gracefully.

### 2. **Test: P1 Channel Identification with Overlapping Deadbands**
**Scenario:** Multiple P0 negative spaces overlap, creating fragmented safe channels.
**Expected:** Room should identify all disjoint safe regions, not just the largest contiguous one.
**Rationale:** In geometric constraint systems (constraint-theory-core), deadbands can create complex safe zones.

### 3. **Test: P2 Optimization with Non-Convex Safe Region**
**Scenario:** Safe channel is non-convex (e.g., donut-shaped), making linear optimization invalid.
**Expected:** Room should use appropriate optimization method for region shape or fall back to centroid.
**Rationale:** Real geometric constraints often produce non-convex regions; optimization must adapt.

### 4. **Test: Protocol Reset After Catastrophic P0 Violation**
**Scenario:** Agent violates P0 so severely that no safe channels exist in current state.
**Expected:** Room should reset to initial state and log "catastrophic violation" rather than hanging.
**Rationale:** From fleet doctrine: when you hit rocks (P0), you must reset and remap.

### 5. **Test: Multi-Agent Constraint Propagation**
**Scenario:** Two agents in same room submit constraints that affect each other's safe channels.
**Expected:** Room should propagate constraints between agents and recalculate all safe channels.
**Rationale:** Fleet context mentions multi-agent simulations; rooms should handle interaction.
