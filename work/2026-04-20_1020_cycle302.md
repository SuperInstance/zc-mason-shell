## Proposed Edge-Case Tests
Based on the Deadband Protocol and room logic, here are 5 additional edge-case tests:

### 1. Test: Agent Attempts P2 Optimization Without P1 Channel
**Scenario:** An agent submits a tile with `priority=2` but the room has no recorded P1 safe channel for that agent.
**Expected:** Tile rejected with error "P1 channel not established".
**Rationale:** Enforces strict P0→P1→P2 progression.

### 2. Test: Concurrent Tile Submission from Same Agent
**Scenario:** Simulate two threads/processes submitting tiles for the same agent ID simultaneously.
**Expected:** Room handles concurrency (e.g., via locking) without corrupting internal state (agent_channels, tile_log).
**Rationale:** Real fleet environments may have concurrent submissions.

### 3. Test: Malformed Tile Payload (Missing Required Fields)
**Scenario:** Submit a tile dict missing the `content` field or with a non-string `content`.
**Expected:** Tile validation fails before protocol checks, returns appropriate error.
**Rationale:** Defensive validation against malformed data.

### 4. Test: Agent Submits P0 Tile That Contradicts Existing P0 Knowledge
**Scenario:** Agent submits a P0 tile stating "avoid X", but the room already has a P0 tile from another agent stating "X is safe".
**Expected:** Room detects contradiction, may reject tile or trigger a conflict resolution (e.g., higher confidence wins, tile flagged for review).
**Rationale:** Maintains consistency of negative space mapping.

### 5. Test: Room Capacity / Memory Limits
**Scenario:** Submit a large number of tiles (e.g., 10,000) to test memory/performance impact.
**Expected:** Room remains responsive; may implement pruning (e.g., keep only latest N tiles per agent) or archiving.
**Rationale:** Ensures room scales with fleet activity.
