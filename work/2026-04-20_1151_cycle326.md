## Edge-Case Test Proposals
Based on the DeadbandRoom implementation and fleet doctrine, here are 5 edge-case tests:

### 1. Test: Room Initialization with Minimal Valid Configuration
**Scenario:** DeadbandRoom is instantiated with only the required parameters (room_id, preset_name). Optional parameters like `max_tiles` or `protocol_config` are omitted.
**Purpose:** Verify defaults are applied correctly and room enters a valid initial state (P0).
**Edge:** Missing optional configuration.

### 2. Test: Tile Submission with Malformed Priority Field
**Scenario:** A tile is submitted where the `priority` field is a string ("P0") instead of an integer (0), or is out of bounds (e.g., -1, 3).
**Purpose:** Ensure the room's validation logic rejects malformed priorities or gracefully maps them according to protocol rules.
**Edge:** Non-standard priority values.

### 3. Test: Concurrent Tile Submissions from Multiple Agents
**Scenario:** Simulate two agents submitting tiles simultaneously (using threading or async mocks). The room's internal queue or state must handle contention.
**Purpose:** Verify thread safety or proper locking mechanisms in `submit_tile`.
**Edge:** Race conditions in shared state.

### 4. Test: Protocol Regression from P2 to P0 Due to Invalid Tile
**Scenario:** Room is in P2 (optimization). A tile is submitted that violates a P0 constraint (e.g., suggests a "rock" action). According to doctrine, the room should regress to P0.
**Purpose:** Test strict priority enforcement and state regression logic.
**Edge:** Protocol state transitions triggered by tile content.

### 5. Test: Training Step with Empty Tile Queue
**Scenario:** Call `train_step()` when the tile queue is empty. This may happen after initialization or if tiles are consumed faster than they are submitted.
**Purpose:** Ensure the room handles empty queues without crashing—either returning a neutral state or logging appropriately.
**Edge:** No data to train on.
