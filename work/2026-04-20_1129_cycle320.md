## Proposed Edge-Case Tests
Based on the DeadbandRoom implementation and fleet doctrine, here are 5 edge-case tests:

### 1. Test: P0 Violation with Ambiguous Input
**Scenario:** Agent submits a tile where the P0 "negative space" mapping is ambiguous (e.g., "avoid risk" without specifics).
**Expected:** Room should reject tile with specific error about ambiguous P0 mapping.
**Rationale:** Ensures P0 mappings are concrete and actionable.

### 2. Test: P1 Channel Exhaustion
**Scenario:** Agent iterates through all possible P1 channels in a single session, reaching "no safe channels" state.
**Expected:** Room should enter a fallback state (e.g., request human oversight) rather than crash.
**Rationale:** Tests boundary condition where deadband protocol finds zero safe operating channels.

### 3. Test: Concurrent Tile Submissions
**Scenario:** Multiple agents (simulated via threads) submit tiles to the same room instance simultaneously.
**Expected:** Room should handle concurrent access without state corruption or race conditions.
**Rationale:** DeadbandRoom may operate in multi-agent environments; concurrency safety is critical.

### 4. Test: Malformed Tile JSON Structure
**Scenario:** Agent submits a tile with valid JSON but missing required fields (e.g., `phase`, `p0_map`).
**Expected:** Room should reject with structured error indicating missing fields, not generic JSON error.
**Rationale:** Improves debugging for agent developers; ensures validation is field-aware.

### 5. Test: P2 Optimization with Zero-Gradient
**Scenario:** Agent identifies a safe P1 channel but the P2 optimization yields zero improvement (flat landscape).
**Expected:** Room should accept tile but log warning about zero-gradient optimization.
**Rationale:** Tests optimization boundary condition; ensures system handles non-improving paths gracefully.
