## Identified Edge Cases & New Tests
Based on the Deadband Protocol's strict priority rules and the room's role in training tile validation, I propose these 5 edge-case tests:

### 1. Test: `test_concurrent_tile_submission`
**Scenario:** Multiple agents submit tiles simultaneously to the same room.
**Edge:** Race conditions in tile validation and state updates.
**Check:** Room maintains correct tile count and doesn't corrupt state.

### 2. Test: `test_p0_empty_negative_space`
**Scenario:** Agent submits a P0 report with an empty "negative space" map.
**Edge:** Empty P0 data—should this be rejected or treated as "no constraints found"?
**Check:** Room's validation logic handles empty dict/list appropriately.

### 3. Test: `test_p2_without_p1`
**Scenario:** Agent attempts P2 optimization without first establishing P1 safe channels.
**Edge:** Direct P2 submission violates protocol's strict priority.
**Check:** Room rejects tile with appropriate error, not just generic violation.

### 4. Test: `test_tile_size_limit`
**Scenario:** Agent submits an extremely large tile (e.g., 10MB JSON).
**Edge:** Memory/processing limits in tile validation.
**Check:** Room gracefully rejects or truncates oversized tiles.

### 5. Test: `test_partial_protocol_sequence`
**Scenario:** Agent submits P0, then P2 (skipping P1), but in separate tiles.
**Edge:** Protocol enforcement across multiple tile submissions from same agent.
**Check:** Room tracks agent's protocol state across submissions.
