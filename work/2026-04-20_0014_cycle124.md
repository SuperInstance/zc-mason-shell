## Edge-Case Test Proposals
Based on the Deadband Protocol doctrine (P0 → P1 → P2 strict priority) and room behavior, I identify these edge cases:

### 1. **Priority Ordering Under Concurrent Writes**
**Scenario:** Multiple tiles added rapidly with mixed priorities (P0, P1, P2).
**Test:** Verify that `get_tiles()` always returns tiles in strict priority order (P0 first, then P1, then P2) regardless of insertion order.
**Rationale:** Ensures protocol adherence during high-load scenarios.

### 2. **Tile Deduplication by Content Hash**
**Scenario:** Adding identical tile content twice (same hash).
**Test:** Check that duplicate tiles are rejected or ignored to prevent training data pollution.
**Rationale:** Prevents redundant tiles from skewing training distributions.

### 3. **Memory Pressure Near Capacity**
**Scenario:** Room at 999 tiles (near max capacity of 1000).
**Test:** Add a P0 tile when room is almost full, verify it succeeds (P0 takes precedence), potentially evicting a lowest-priority tile if needed.
**Rationale:** Tests priority-based eviction policy under memory constraints.

### 4. **Malformed Priority Field Types**
**Scenario:** Tile with priority as string ("P0") instead of integer (0).
**Test:** Ensure room rejects tiles with non-integer priority values.
**Rationale:** Type safety prevents parsing errors during training.

### 5. **Empty Room Query Behavior**
**Scenario:** Calling `get_tiles()` on a newly initialized empty room.
**Test:** Verify it returns empty list (not None or raising exception).
**Rationale:** Ensures robust API behavior for downstream consumers.
